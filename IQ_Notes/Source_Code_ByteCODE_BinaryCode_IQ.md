# Source Code vs Bytecode vs Binary Code

## Example: `01_HelloWorld.js`

**Source File:** `01_chapter_Javascript/01_HelloWorld.js`

```js
console.log("Hello, this is vishnu's PC");
```

---

## Comparison Table

| Aspect | Source Code | Bytecode | Binary Code |
|--------|------------|----------|-------------|
| **Definition** | Human-readable code written in a high-level language (JavaScript) | Intermediate representation — partially compiled, platform-independent | Machine-level instructions in 0s and 1s, directly executed by the CPU |
| **Who/What Reads It** | Developers read and write it | A virtual machine or runtime engine (e.g., V8's Ignition interpreter, JVM, .NET CLR) | The CPU / processor directly |
| **How It Looks** | `console.log("Hello, this is vishnu's PC");` | Compact numeric opcodes, e.g., `LdaGlobal [0]`, `Star r0`, `Ldar r0`, `CallProperty1` (V8 bytecode) | Raw binary: `01001010 11000011 00101011 ...` or hex: `48 65 6C 6C 6F ...` |
| **Readability** | Easily readable by humans | Hard for humans; readable only with disassemblers/tools | Impossible for humans to read directly |
| **Portability** | Portable — same code runs wherever the runtime exists | Portable — same bytecode runs on any platform with the matching VM | **Not portable** — compiled for a specific CPU architecture (x86, ARM, etc.) |
| **Generation** | Written by the programmer | Generated by the compiler/interpreter (e.g., V8 compiles JS → bytecode) | Generated by a compiler or assembler, or produced by JIT from bytecode |
| **Execution** | Cannot run directly — must be compiled or interpreted first | Executed by a virtual machine (e.g., V8 Ignition, JVM, .NET CLR) | Executed directly by the CPU's instruction pipeline |
| **For `01_HelloWorld.js`** | The file you wrote: a single `console.log(...)` statement | V8 parses JS → AST → Ignition generates bytecode (opcodes like `LdaGlobal`, `CallProperty1`) | V8's JIT compiler (TurboFan/Maglev) further compiles hot bytecode into native x86/ARM machine code |
| **File Extension** | `.js` | Depends on platform: `.pyc` (Python), `.class` (Java), `.dll`/`.exe` (C# IL). JS bytecode is in-memory only (V8 does not save it). | `.exe`, `.bin`, `.o`, `.so`, `.dll` (native compiled code) |
| **Debugging** | Easy — line-by-line in IDE, source maps point here | Harder — need VM-level debugging tools (e.g., `--print-bytecode` in Node.js) | Very hard — requires disassemblers (gdb, objdump, IDA Pro) |

## Summary Flow for `01_HelloWorld.js`

```
Source Code (.js)
    │
    │  You write this
    │  console.log("Hello, this is vishnu's PC");
    ▼
   Parsing (V8 Parser)
    │
    │  Converts text → AST (Abstract Syntax Tree)
    ▼
   Bytecode (V8 Ignition Interpreter)
    │
    │  Generates platform-independent bytecode
    │  e.g., LdaGlobal [console], Star r0, Ldar r0, CallProperty1 [log], ...
    ▼
   Execution (V8 Ignition VM)
    │
    │  Interprets bytecode line by line
    │  ⚡ Hot code paths detected →
    ▼
   Binary / Machine Code (V8 TurboFan / Maglev JIT)
    │
    │  JIT-compiles frequently-run bytecode into native x86/ARM instructions
    │  e.g., mov rax, [rip+offset], call printf, ...
    ▼
   CPU executes native binary instructions directly
```

---

## One-Liner Summary

- **Source Code** = What you write (`.js` file)
- **Bytecode** = What the VM runs (intermediate, portable)
- **Binary Code** = What the CPU runs (native, architecture-specific)
